# Технологический стек — Hunar MVP

> `Hunar` — рабочее кодовое имя продукта (плейсхолдер).
> Документ фиксирует выбор технологий, обоснование и правила их использования.

---

## 1. Обзор

Hunar — двусторонний маркетплейс найма для IT-рынка Узбекистана. Стек выбран под одну цель: **быстро собрать безопасный MVP силами небольшой команды**, без преждевременной инженерии.

Принципы выбора:
- **Монолит, не микросервисы.** На масштабе MVP микросервисы — самонаказание.
- **BaaS вместо своего бэкенда.** Supabase закрывает БД, auth и storage из коробки.
- **Безопасность данных встроена в стек (RLS), а не прикручена сбоку.**
- **Никакого «на будущее».** Elasticsearch, ML, Kubernetes — вне MVP.

## 2. Стек одним взглядом

| Слой | Технология |
|---|---|
| Frontend / SSR | Next.js (App Router, TypeScript) |
| UI / стили | Tailwind CSS |
| Анимации | Motion (framer-motion) |
| БД | PostgreSQL (через Supabase) |
| Backend / API | Supabase + Next.js Route Handlers / Server Actions |
| Auth | Supabase Auth — Google OAuth + GitHub OAuth |
| Хранилище CV | Supabase Storage (приватный bucket + RLS) |
| Публичные изображения | Next.js Route Handler → imgbb (или публичный bucket Supabase) |
| Поиск | PostgreSQL Full-Text Search + `pg_trgm` |
| Валидация | Zod |
| Хостинг фронта | Vercel (или аналог с SSR) |

## 3. Frontend / Backend — Next.js

- **App Router**, **TypeScript** строго.
- **Server Components по умолчанию**; `'use client'` — только для интерактива (формы, табы, ховеры, поиск-бар).
- **Server Actions / Route Handlers** — для мутаций и серверной логики (валидация + обращение к Supabase).
- **SSR/SSG** для публичных страниц (`/`, `/jobs`, `/jobs/[slug]`, `/companies/[slug]`) — критично для SEO.
- Два Supabase-клиента: серверный и браузерный. **`service_role`-ключ — только в серверном коде**, никогда не в браузерном бандле.

## 4. Supabase (БД + Auth + Storage)

### PostgreSQL
- Основная и единственная БД.
- Расширения: `pg_trgm` (нечёткий поиск), GIN-индексы по массиву `skills`.
- Миграции хранятся в репозитории (Supabase CLI).

### Row Level Security (RLS) — обязательно
- RLS включён на **всех** таблицах с пользовательскими данными.
- Политики пишутся вместе со схемой, до доступа из приложения.
- Ключевые ограничения: кандидат видит только своё; работодатель — отклики только по своим вакансиям; контакт кандидата открывается только при одобренном отклике; публично видны только вакансии `status = approved`.

### Auth
- **Google OAuth** (основной) + **GitHub OAuth** (для IT-аудитории).
- Без email/пароля и без SMS в MVP.
- При первом входе — onboarding: роль + согласие на обработку ПД.

### Storage
- **CV** → приватный bucket, доступ по Storage-политике (владелец + работодатель с одобренным откликом). **Никогда не публичный.**
- Публичные изображения (лого, аватарки) → публичный bucket Supabase **или** imgbb.

## 5. Стили — Tailwind CSS

- Utility-first, без отдельной дизайн-системы на старте.
- Mobile-first (основной трафик — мобильный).
- Консистентность через конфиг Tailwind (палитра, типографика, отступы).

## 6. Анимации — Motion (framer-motion)

- Сдержанно: появление секций при скролле, микро-ховеры, лёгкий hero.
- **Не блокировать первый рендер**, не ронять перформанс на мобильных.
- Уважать `prefers-reduced-motion`.

## 7. Поиск и матчинг

- **Поиск:** PostgreSQL FTS + `pg_trgm` + GIN по `skills`. **Без Elasticsearch** — преждевременно.
- **Матчинг:** rule-based скоринг (навыки + формат/локация + вилка + грейд) как подсказка админу. Финальный подбор — вручную. **Без ML.**

## 8. Структура проекта

```
app/
  (public)/            # лендинг, /jobs, /jobs/[slug], /companies
  (auth)/              # /login, /onboarding
  candidate/           # /profile, /applications, /settings
  employer/            # /employer/*
  admin/               # /admin
  api/                 # Route Handlers (напр. проксирование загрузки изображений)
components/            # переиспользуемый UI
lib/

  validation/          # zod-схемы
types/                 # общие типы
```

## 10. Переменные окружения

```
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=
SUPABASE_SERVICE_ROLE_KEY=        # только сервер, не в браузер
IMGBB_API_KEY=                    # только сервер (если используется imgbb)
NEXT_PUBLIC_SITE_URL=
```

- Секреты — только через окружение, никогда в коде.
- `service_role` и `IMGBB_API_KEY` — исключительно в серверном коде / Route Handlers.

## 11. Чего в стеке НЕТ (и почему)

| Технология | Почему не сейчас |
|---|---|
| Микросервисы | Нет масштаба; усложняют без выгоды |
| Elasticsearch | Postgres FTS достаточно для MVP |
| ML-матчинг | Преждевременно; rule-based + человек |
| Kubernetes | Избыточно; managed-хостинг |
| Redis | Не нужен на старте |
| Отдельный бэкенд (NestJS/Go) | Supabase + Next.js закрывают потребности MVP |
| Платёжный SDK | Биллинг вне MVP (работодатели вручную) |
| Email/SMS-провайдеры | Уведомления вне MVP |

## 12. Важные оговорки (риски стека)

- **imgbb — только для публичных изображений.** CV туда лить нельзя: публичные ссылки + персональные данные = нарушение закона о ПД. Резюме — только в приватном Supabase Storage с RLS.
- **Supabase хостится за рубежом (AWS, нет UZB-региона).** Для обычных персданных это допустимо после смягчения закона о ПД (с 27.03.2026) при выполнении условий. Требуется: регистрация базы ПД в Государственном реестре + подтверждение условий хранения. **Согласовать с юристом до запуска.**
- **RLS — главная точка отказа по безопасности.** Политики для маркетплейса (разный доступ у кандидата/работодателя/админа) легко настроить с дырой и слить данные. Каждую политику тестировать на «чужих» аккаунтах отдельно.

## 13. Эволюция стека после MVP (для контекста)

- Платежи: Payme / Click / Uzum (локальные провайдеры, не Stripe).
- Уведомления: SMS через Eskiz.uz / Play Mobile + email.
- Phone-OTP auth — при выходе за пределы IT-аудитории.
- Встроенный чат, мобильное приложение (React Native / PWA), при росте поиска — Meilisearch/Typesense.
