# Техническое задание — Hunar MVP

**Продукт:** Hunar — нишевый маркетплейс найма для IT-рынка Узбекистана
**Версия документа:** 1.0 (MVP)
**Статус названия:** `Hunar` — рабочее кодовое имя (плейсхолдер)

---

## 1. Цель и суть продукта

Hunar связывает IT-кандидатов и работодателей в Узбекистане. Ключевое отличие от существующих площадок (hh.uz и др.) — **курация и качество вместо объёма и спама**: вакансии проходят модерацию, профили могут верифицироваться, зарплаты показываются прозрачно в **UZS и USD**.

В MVP площадка выглядит как полноценный маркетплейс, но подбор на старте выполняется вручную через админку (concierge-режим) — это сознательный обход проблемы холодного старта двустороннего рынка.

**Цель MVP:** довести до первого реального найма (matched & hired) в одной нише, проверив, что работодатели возвращаются постить вакансии повторно.

## 2. Роли пользователей

| Роль | Описание |
|---|---|
| **Кандидат** | Соискатель. Создаёт профиль, откликается на вакансии. |
| **Работодатель** | Представитель компании. Постит вакансии, смотрит отклики. |
| **Админ / Ops** | Модерация вакансий, верификация компаний и кандидатов, ручной подбор, борьба со спамом. В MVP — скрытый движок матчинга. |

## 3. Технологический стек

- **Next.js** (App Router, TypeScript, Server Components)
- **Supabase** — PostgreSQL + Auth + Storage + Row Level Security
- **Tailwind CSS** — стили
- **Motion (framer-motion)** — анимации, сдержанно
- **Аутентификация:** Google OAuth + GitHub OAuth (без email/пароля и SMS)
- **CV:** Supabase Storage, приватный bucket с RLS
- **Публичные изображения** (лого, аватарки): Next.js Route Handler → imgbb

## 4. Функциональные требования

### 4.1 Кандидат
- Регистрация/вход через Google или GitHub.
- Согласие на обработку персональных данных при регистрации (обязательно).
- Создание и редактирование профиля: ФИО, заголовок, навыки (теги), грейд (junior/middle/senior/lead), годы опыта, город, формат (remote/onsite/hybrid), уровень английского, ожидания по зарплате (мин/макс, UZS и USD).
- Загрузка файла CV (PDF/DOCX) в приватное хранилище.
- Поиск и фильтрация вакансий.
- Отклик на вакансию (опц. короткая сопроводительная заметка).
- Согласие на раскрытие контакта при отклике.
- Просмотр своих откликов и их статусов.

### 4.2 Работодатель
- Регистрация/вход через Google или GitHub.
- Создание профиля компании: название, логотип, описание, сайт, признак «резидент IT Park» (опц.).
- Создание вакансии: заголовок, описание, требуемые навыки, грейд, формат, локация, вилка зарплаты (валюта UZS/USD), статус.
- Вакансия уходит на модерацию перед публикацией.
- Просмотр откликов по своей вакансии.
- Смена статуса отклика: новый → просмотрен → шортлист → одобрен / отказ.
- При статусе «одобрен» — доступ к контакту кандидата (после согласия кандидата).

### 4.3 Админ / Ops
- Очередь модерации вакансий (approve/reject с причиной).
- Верификация компаний (бейдж «проверено»).
- Верификация кандидатов (ручной бейдж «verified» — без автотестов в MVP).
- Просмотр rule-based матч-подсказок и ручной подбор (свести кандидата и вакансию).
- Бан/флаг спама и нарушителей.

## 6. Модель данных (Supabase / PostgreSQL)

> Все таблицы с пользовательскими данными — с включённым RLS и явными политиками.

- **profiles** — `id (uuid, fk auth.users)`, `role (candidate/employer/admin)`, `full_name`, `avatar_url`, `consent_pd_at`, `created_at`
- **candidate_profiles** — `user_id (fk)`, `headline`, `bio`, `skills (text[])`, `grade`, `years_exp`, `city`, `work_format`, `english_level`, `salary_min`, `salary_max`, `salary_currency`, `cv_path (storage)`, `verification_status`, `contact_telegram`, `contact_phone`
- **companies** — `id`, `owner_id (fk)`, `name`, `slug`, `logo_url`, `description`, `website`, `is_itpark_resident`, `is_verified`, `created_at`
- **jobs** — `id`, `company_id (fk)`, `title`, `slug`, `description`, `skills (text[])`, `grade`, `work_format`, `city`, `salary_min`, `salary_max`, `salary_currency`, `status (draft/pending/approved/rejected)`, `reject_reason`, `created_at`
- **applications** — `id`, `job_id (fk)`, `candidate_id (fk)`, `cover_note`, `status (new/viewed/shortlist/approved/rejected)`, `contact_disclosed (bool)`, `created_at`
- **moderation_logs** — `id`, `entity_type`, `entity_id`, `admin_id`, `action`, `reason`, `created_at`

**RLS — ключевые правила:**
- Кандидат видит/редактирует только свой `candidate_profile` и свои `applications`.
- Работодатель видит `applications` только по своим `jobs`.
- Контакт кандидата (`contact_*`) виден работодателю только если `applications.status = approved` И `contact_disclosed = true`.
- Публично видны только `jobs` со `status = approved`.
- Полный доступ — только роль `admin` (через серверный код с `service_role`, не на клиенте).

## 7. Аутентификация

- Только OAuth: **Google** (основной) + **GitHub** (для IT-аудитории).
- При первом входе — `/onboarding`: выбор роли + обязательное согласие на обработку ПД (пишется `consent_pd_at`).
- Без email/пароля и без SMS в MVP. (Примечание: для расширения за пределы IT-аудитории позже потребуется phone-OTP.)

## 8. Хранилище файлов

- **CV** → приватный bucket Supabase Storage. Доступ по Storage-политике: владелец + работодатель с одобренным откликом. Никогда не публичный, никогда не на imgbb.
- **Логотипы компаний, аватарки** → публичные изображения через Next.js Route Handler → imgbb (или, предпочтительно, тоже Supabase Storage публичный bucket, чтобы убрать лишнюю зависимость).

## 9. Поиск и матчинг

- **Поиск (v1):** PostgreSQL full-text search + `pg_trgm` + GIN-индекс по `skills`. Без Elasticsearch.
- **Матчинг (v1):** rule-based скоринг = пересечение навыков + совпадение формата/локации + пересечение вилки зарплаты + соответствие грейда. Результат — подсказка админу в `/admin`. **Финальный подбор делает человек.** ML — вне скоупа.

## 10. Коммуникация сторон (вместо чата)

Встроенного чата в MVP нет. Сценарий:
1. Кандидат откликается → даёт согласие на раскрытие контакта.
2. Работодатель ревьюит отклик, ставит статус «одобрен».
3. После одобрения обе стороны видят контакт друг друга (Telegram/телефон), `contact_disclosed = true`.
4. Дальнейшее общение — вне платформы (Telegram — стандарт рабочей переписки в Узбекистане).

## 11. Нефункциональные требования

- **Закон о персональных данных (Узбекистан):** база ПД регистрируется в Государственном реестре; согласие пользователя фиксируется; обеспечивается право на удаление. Размещение Supabase за рубежом для обычных персданных допустимо после смягчения закона (с 27.03.2026) при выполнении условий — **подтвердить с юристом до запуска.** CV и контакты — под усиленным контролем доступа (RLS).
- **SEO:** SSR/SSG для публичных страниц, `schema.org/JobPosting`, корректные meta/OpenGraph, sitemap, человекочитаемые `slug`.
- **Локализация:** интерфейс RU + UZ. Англо-CV кандидата — отдельная сущность.
- **Производительность:** публичные страницы отдаются с сервера; индексы под поиск; изображения оптимизированы.
- **Безопасность:** RLS на всех пользовательских таблицах; `service_role` только на сервере; серверная валидация (zod); защита от спама в формах.
- **Адаптивность:** mobile-first (основной трафик мобильный).

## 12. Вне скоупа MVP

Встроенный чат · email/SMS/push-уведомления · платежи и биллинг · автоматические тесты навыков · ML-матчинг · нативное приложение · полный английский интерфейс · резюме-конструктор · бенчмарки зарплат · рекомендательная лента · saved jobs.

## 13. Критерии приёмки MVP

- [ ] Кандидат может зарегистрироваться (Google/GitHub), заполнить профиль, загрузить CV, найти и откликнуться на вакансию.
- [ ] Работодатель может создать компанию, запостить вакансию, та проходит модерацию и публикуется.
- [ ] Работодатель видит отклики, меняет статусы; при «одобрено» получает контакт кандидата.
- [ ] Админ модерирует вакансии, верифицирует, видит матч-подсказки, делает ручной подбор, банит спам.
- [ ] RLS закрывает доступ к чужим данным (проверено тестами на «чужих» аккаунтах).
- [ ] CV недоступен публично и по прямой ссылке посторонним.
- [ ] Публичные страницы вакансий индексируются и содержат корректную разметку.
- [ ] Интерфейс работает на мобильном и десктопе.

## 14. Этапы (ориентировочно)

1. **Скелет + схема БД + RLS + auth** — основа, политики доступа, OAuth, onboarding с согласием.
2. **Кандидат** — профиль, CV, каталог вакансий, поиск, отклик.
3. **Работодатель** — компания, постинг вакансий, отклики, статусы, раскрытие контакта.
4. **Админка** — модерация, верификация, матч-подсказки, ручной подбор, бан.
5. **Лендинги + SEO + юр. страницы + полировка + проверка RLS/безопасности.**

> Оценка сроков: при команде из 3 миддлов, знакомых со стеком, и зафиксированном скоупе — code-complete ≈ 7–9 рабочих недель (без чата). До реального запуска (с юр. оформлением и засеянными вакансиями) — ≈ 3 месяца.
